# Whisper Call v0.2 Candidate Research

**Date:** 2026-04-26
**Goal:** Evaluate three v0.2 candidates and recommend the highest-leverage scope.

## Candidates

### 1. MCP-subprocess hang fix (`_mcp_call`)

**Location:** `C:\Users\kas41\.claude\plugins\whisper-call\scripts\call_driver.py:31-55`

**Root cause analysis (read of code):**
- `subprocess.run(input=json.dumps(request)+"\n", capture_output=True, text=True, timeout=60)` writes the request and closes stdin.
- The `@modelcontextprotocol/sdk` stdio transport reads stdin via Node streams. On stdin EOF, the SDK does not call `server.close()` automatically — the process stays alive waiting for more JSON-RPC frames.
- Result: subprocess always runs to the 60s timeout before returning. Effective bug: every CLI driver call that touches MCP is unusably slow.
- Slash commands work because Claude Code orchestrates the MCP via its own long-lived stdio connection — it never invokes our CLI driver's `_mcp_call`.

**Fix options (ranked):**
1. **Use `subprocess.Popen` with manual lifecycle** (recommended)
   - Write request, read stdout line-by-line until first valid JSON-RPC response, then `proc.terminate()`.
   - ~30 lines of Python. Isolated to one function. Test with mock subprocess.
2. **Wrap MCP in a request/exit shim** (more invasive)
   - Modify MCP server to detect a `meta.exit_after_response` flag and self-terminate.
   - Touches both Python and TypeScript codebases.
3. **Long-lived subprocess pool** (more complex)
   - Cache one MCP subprocess per CLI driver run, reuse across calls.
   - Adds complexity for a CLI driver that is currently called once per invocation.

**Effort:** LOW (1-2 hours)
**Value:** HIGH — unblocks all CLI driver MCP paths (trigger, set-mode, get-mode, macro CRUD)
**Risk:** LOW — isolated change, easy to test
**Blast radius:** Single function in single file

### 2. Template auto-fill

**Goal:** Replace `[FILL]` placeholders in templates with values inferred from context (recent messages, project state).

**Effort:** MEDIUM-HIGH (4-8 hours)
- Need a context extractor (recent conversation, file content, git diff)
- Need an LLM call to infer fills per placeholder
- Need integration with template injection path
- Settings: opt-in flag, fill confidence threshold, etc.

**Value:** MEDIUM — convenience improvement; templates are usable today with manual fill
**Risk:** MEDIUM
- LLM calls add latency, failure modes, and cost
- Auto-fill may produce wrong values, requiring user review (defeats hands-free goal)
- Spec explicitly listed this as **out of scope for v0.1** — author intent was to defer

**Verdict:** DEFER. Spec said v1 leaves placeholders manual. Auto-fill is a nice-to-have, not a blocker.

### 3. Concurrent invocation queueing

**Goal:** Per spec §11 — handle concurrent `/call` invocations FIFO with max queue depth 3.

**Effort:** MEDIUM (3-5 hours)
- Add a request queue in MCP `state.ts`
- FIFO processing with depth check
- Drop-with-warning beyond depth 3
- Integration with `trigger_dictation` flow
- Concurrency tests (timing-sensitive — flaky risk)

**Value:** LOW-MEDIUM
- In practice, users rarely fire `/call` concurrently
- Wispr Flow itself prevents concurrent dictations (single audio capture)
- The race is more theoretical than practical

**Risk:** MEDIUM
- Concurrency bugs are hard to spot in tests
- May introduce non-determinism that's hard to debug

**Verdict:** DEFER. Real-world concurrent /call is rare; benefit/risk skews against shipping now.

## Recommendation

**v0.2 scope: MCP-hang fix only (Candidate 1).**

Rationale:
- Single, well-bounded fix
- High value (unblocks broken functionality)
- Low effort + low risk
- Can ship as v0.2.0 within hours

Defer Candidates 2 and 3 to v0.3+ when:
- Real users report templates frustrating without auto-fill (Candidate 2)
- Concurrency complaints emerge (Candidate 3)

- L2 (cross-module): End-to-end with actual MCP subprocess — `python call_driver.py get-mode` returns within 2s
